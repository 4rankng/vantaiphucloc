### Finding 1

```
Type: Bug
Layer: Frontend
Affected Role/Flow: ketoan — Settings / Nhà thầu (Vendors)
Severity: Medium
Description: The "Nhà thầu" (Contractors/Vendors) list at /accountant/settings/vendors shows all company
vendors with "Loại: Cá nhân" (Individual type). The same companies (Công ty TNHH HAP, Công ty TNHH
HẢI AN, Công ty TNHH NEWWAY, Công ty TNHH PAN HẢI AN) correctly show "Loại: Công ty" on the
Khách hàng (Clients) page, but show "Loại: Cá nhân" on the Nhà thầu page.
Recommendation: Apply the same isCompany heuristic fix that was applied to the Clients page (task-0059)
to the Vendors/Contractors page. The fix likely lives in the vendor list component — check for the Loại
column rendering logic and apply the same "Công ty" vs "Cá nhân" detection.
```

Location: `/accountant/settings/vendors`
Observation: 5 vendor rows shown. All 4 company vendors (Công ty TNHH ...) display "Cá nhân" in the Loại column. Only "Vận Tải Phúc Lộc" also shows "Cá nhân" (which may be correct for that entity).
Impact: Misleads accountants who manage contractor relationships. Broken data display undermines trust in the system and makes it impossible to distinguish company vs. individual contractors at a glance.

### Finding 3

```
Type: Usability Issue
Layer: Frontend
Affected Role/Flow: ketoan — Ghép chuyến (Work Orders matching) — manual match confirmation
Severity: High
Description: When a user clicks "Ghép" on a low-confidence match (score 3/6 or lower), the action
completes immediately with NO confirmation dialog and NO success/error toast. The only feedback is
that the work order disappears from the "chờ ghép" count (20 → 19). This is a data-loss risk: an
accidental click or misclick permanently matches two orders with no undo prompt.
Recommendation: For scores < 5/6, show a confirmation dialog before completing the match. Separately,
always show a success toast ("Đã ghép thành công") or error toast on completion regardless of score.
This was previously reported as OPEN-01. Flagging again as High because no progress observed.
```

Location: `/accountant/work-orders` (match panel)
Observation: W001015 had score 3/6. Clicked Ghép → immediate action, no dialog, no toast. Work order count changed from 20 to 19.
Impact: Accidental matches cannot be caught before commit. With no undo workflow, a wrong match requires manual correction, which is time-consuming and error-prone.

### Finding 4

```
Type: Usability Issue
Layer: Frontend
Affected Role/Flow: ketoan — Ghép chuyến — auto-match
Severity: Medium
Description: Clicking "Tự động ghép" provides zero user feedback. No loading spinner, no success
toast, no result summary (e.g., "Đã tự động ghép 0 chuyến" or "Đã ghép 3 chuyến"). The user cannot
tell whether the action ran, failed, or found nothing to match.
Recommendation: Add (a) a loading indicator while the API call is in flight, and (b) a toast on
completion: success with count matched, or an explicit "Không tìm thấy cặp ghép nào" message if 0.
This was OPEN-03. Flagging again as it remains entirely unaddressed.
```

Location: `/accountant/work-orders` (top bar)
Observation: Clicked "Tự động ghép" → waited 2 seconds → 0 toasts, 0 dialogs, 0 loading indicators. Count remained at 19.
Impact: Users will click the button multiple times thinking it did not respond, potentially causing duplicate processing or confusion about system state.

### Finding 5


Impact: Work orders with unusual routes or data entry discrepancies become permanently unmatched in the UI until an admin manually corrects the underlying data.

### Finding 6

## UX/UI Findings

---

### UX-01
**Severity:** 🔴 Critical
**Location:** `/accountant/trips` — Doanh thu column
**Observation:** Every one of the 19 trip orders in the Đơn hàng list displays `0 ₫` in the Doanh thu column. The KPI card on the same page correctly shows 9 Chờ khớp + 10 Đã khớp but Doanh thu is universally zero. The dashboard KPI shows `DOANH THU (ĐƠN HÀNG THÁNG): 25.050.000 ₫` — confirming the data exists in aggregate but the per-row figure is broken.
**Impact:** The Doanh thu column is the primary financial signal for the kế toán's reconciliation decision. All-zero values make the column useless and erode trust in the entire page's data accuracy. Operators will not know which orders generate revenue.
**Recommendation:** Verify that `GET /api/v1/trip-orders` response includes a `price` or `revenue` field. If the field was renamed or removed in the current deploy, update the table column mapping. If pricing rules are not applied at import time, compute the revenue from the linked pricing table on the backend.

---

### UX-02
**Severity:** 🔴 Critical
**Location:** `/director` — KPI card "Đã khớp"
**Observation:** The "Đã khớp" KPI counter shows `0` on the director dashboard. The "Lệnh vận chuyển gần đây" panel directly below shows T0032 as "Đã khớp". The counter is wrong while the list is correct — they use different data sources.
**Impact:** The director's primary oversight metric (how many orders are reconciled) is reporting false data. A director making decisions based on "0 reconciliations" when 10+ exist will draw wrong operational conclusions.
**Recommendation:** The KPI counter should use the same data source as the recent list, or query `GET /api/v1/trip-orders?status=matched&date_from=...&date_to=...` and count the results consistently.

### UX-03
**Severity:** 🔴 Critical
**Location:** `/accountant/settings/clients` — LOẠI column
**Observation:** All 4 "Công ty TNHH" companies display "Cá nhân" (Individual) in the LOẠI column of the table. Opening the edit dialog for any of these companies shows "Loại: Công ty" selected — confirming the stored value is correct ("company"). The table-column display is translating `company` → "Cá nhân" instead of "Công ty".
**Impact:** Every listed company appears as "Individual" to the accountant. If ketoan is auditing customer types or generating reports based on company/individual distinction, the displayed data is actively wrong. The table has a reverse enum mapping.
**Recommendation:** In the Khách hàng table column renderer, the mapping should be `{ company: 'Công ty', individual: 'Cá nhân' }`. The current mapping appears inverted: `{ company: 'Cá nhân', individual: 'Công ty' }`.
**Affected file (likely):** `frontend/src/pages/accountant/settings/ClientsPage.tsx` or the DataTablePro column definition for "Loại".

### UX-04
**Severity:** 🟡 Major
**Location:** `/director` — sidebar / navigation
**Observation:** In v5, the director had a sidebar with 3 navigation items (Tổng quan, Thông báo, Quản lý tài khoản). In this build, the director has NO sidebar at all — only a minimal header with "Thông báo" and "Tài khoản" icon-buttons. There is no way to navigate to any other page from the dashboard without knowing the URL.
**Impact:** The director is trapped on the dashboard. The "Quản lý tài khoản" route and any director-specific views are unreachable via the UI. This is a regression from v5.
**Recommendation:** Restore the director sidebar. The `DIRECTOR_MENU` config in `AppSidebar.tsx` should include at minimum: Tổng quan, Thông báo, Quản lý tài khoản.

### UX-05
**Severity:** 🟡 Major
**Location:** All pages — network layer
**Observation:** `POST /api/v1/push/subscriptions → 201` fires 4–5 times per session (observed across ketoan + giamdoc sessions). This was confirmed fixed in v5 (0 POSTs across 689 requests). The regression re-introduces backend noise and potential duplicate push subscription records.
**Impact:** Server-side: duplicate subscription entries per user per session, wasted resources. Client-side: re-registration on every route change suggests a component rendering the push setup hook is being remounted on navigation — this indicates a potential layout/routing architecture issue.
**Recommendation:** The push subscription registration should use a `useEffect` with an empty dependency array (`[]`) that runs once per app session, not per page/component mount. Add a `sessionStorage.getItem('push_registered')` guard to prevent re-registration within the same browser session.

### UX-06
**Severity:** 🟡 Major
**Location:** `/accountant/settings/salary` — Kỳ lương period
**Observation:** The salary configuration shows Từ ngày=26, Đến ngày=25, with the description "Ngày 26 tháng này → ngày 25 tháng sau". The "Kỳ hiện tại" display shows `25/04 → 24/05` — off by one day. Today (2026-05-11) with from=26 should produce kỳ hiện tại = 26/04 → 25/05, not 25/04 → 24/05. The period calculation appears to subtract 1 from the configured start day.
**Impact:** Salary calculations run on the wrong date range. Trips completed on 2026-04-25 would be included in this period (should not be), and trips on 2026-04-26 may be excluded (should be included). Over a month, this could shift salary by up to 1 day of trips.
**Recommendation:** Fix the period calculation to use `configured_from_day` inclusive (not `configured_from_day - 1`). Verify backend salary computation uses the same date range as displayed.

### UX-07
**Severity:** 🟡 Major
**Location:** `/driver/job/:id` — Cung đường field
**Observation:** Every trip detail page for both matched and unmatched work orders shows "Cung đường: -" (route: dash). Route information is critical for drivers to verify they selected the correct trip, yet it is universally absent.
**Impact:** Driver cannot confirm the route on their trip detail. If multiple trips exist for the same customer, the driver has no way to distinguish them by route.
**Recommendation:** The work order data model should store the Điểm lấy and Điểm trả from the creation form. The detail page should display these as "Cung đường: {from} → {to}". This may require backend confirmation that the `route` field is persisted and returned in `/api/v1/work-orders/:id`.

### UX-08
**Severity:** 🟡 Major
**Location:** `/accountant/work-orders` and `/accountant/trips` — filter chip vocabulary
**Observation:** Both pages now use "Chờ khớp" as the filter chip label for unmatched items. In v5, the Đơn hàng page was correctly fixed to "Chờ ghép". This has regressed — the Đơn hàng page now says "Chờ khớp" again. "Đã khớp" is correct on both pages. The Ghép chuyến page previously said "Hoàn thành" (fixed) but both pages now consistently use the wrong "Chờ khớp" instead of "Chờ ghép".
**Impact:** The product vocabulary standard (`Chờ ghép` for unmatched orders, `Đã khớp` for matched) is not applied. Users must mentally re-translate across pages.
**Recommendation:** The correct mapping is `status="pending" → "Chờ ghép"` and `status="matched" → "Đã khớp"` on BOTH pages. Fix in: likely `WorkOrderList.tsx` and `TripOrderList.tsx` filter chip configurations.

### UX-09
**Severity:** 🟡 Major
**Location:** `/driver` — DriverHome trip date display
**Observation:** All trip cards on the driver home show "Hôm nay · 09:16" as the date/time, even for trips with dates from April and early May. Today is 2026-05-11. Trips from April 24–30 show "Hôm nay · 09:16" — clearly wrong. This appears to be the `created_at` timestamp of the seeded data rather than the actual trip execution date.
**Impact:** Drivers cannot tell when a trip occurred from the list view. "Hôm nay" for an April trip is actively misleading.
**Recommendation:** Display the trip's `date` field (the actual execution date), not `created_at`. The format "Hôm nay · HH:MM" should only appear for trips whose date equals today; otherwise use "DD/MM · HH:MM".

### UX-10
**Severity:** 🟡 Major
**Location:** `/director` — Hoạt động gần đây section
**Observation:** The activity feed shows garbled copy:
- "Ghép ghép chuyến #1" (action word "Ghép" duplicated)
- "Tạo reconciliations #2" (English word "reconciliations" leaked from audit log action type)
- "Cập nhật chuyến #1" (generic, no actor identified)
- "Cập nhật đơn hàng #32" (no actor identified)
The underlying data is rendered directly from audit log action strings without translation.
**Impact:** An oversight dashboard with unreadable activity entries fails the director's need to understand what is happening in the system. "Tạo reconciliations" is meaningless to a Vietnamese business user.
**Recommendation:** Map audit log action types to human-readable Vietnamese strings:
- `CREATE_RECONCILIATION` → "Kế toán đã ghép chuyến"
- `UPDATE_WORK_ORDER` → "Cập nhật phiếu chuyến"
- `UPDATE_TRIP_ORDER` → "Cập nhật đơn hàng"
Include the actor name (from `audit_log.user_id → user.name`) in each entry.

### UX-11
**Severity:** 🟢 Minor
**Location:** `/driver/work-orders/new` — auto-fill chips
**Observation:** Clicking a quick-fill route chip (e.g., "PAN / Tân Cảng → Hiệp Phước") fills Điểm lấy and Điểm trả but NOT the Khách hàng dropdown. The chip contains the client code (PAN) which maps to "Công ty TNHH PAN HẢI AN". Drivers must click the chip AND separately select the customer — two actions where one should suffice.
**Impact:** The UX benefit of quick-fill chips is halved. Drivers on mobile will find this frustrating, especially since the customer dropdown requires a separate tap/scroll.
**Recommendation:** Chips should populate all three fields (customer + pickup + dropoff) atomically. The `chipData` object likely has `clientId` — pass it to the customer selection handler.

### UX-12
**Severity:** 🟢 Minor
**Location:** `/accountant/settings/salary` — Kỳ lương redesign
**Observation:** The salary page has been redesigned. "Lịch sử kỳ lương" with per-driver salary cards is gone. Instead there is a "Xem thu nhập tài xế" section with a dropdown to select a driver. This changes the UX from a batch-overview view (see all 3 drivers at once) to a sequential single-driver view.
**Impact:** Ketoan cannot quickly scan all drivers' salaries in one view. The per-driver lookup requires 3 separate interactions to review 3 drivers.
**Recommendation:** If the redesign is intentional, add a summary row or a "Xem tất cả" view that restores the multi-driver overview. If unintentional, restore the historical driver cards.

---

### UX-13
**Severity:** 🟢 Minor
**Location:** `/accountant/settings/drivers` — driver table columns
**Observation:** The Tài xế table now shows only "Tài xế" (Name) and "SĐT" columns. In v5 it showed "Tài xế / SĐT / Biển số xe / Nhà xe" (4 columns). "Biển số xe" (vehicle plate) and "Nhà xe" are now absent from the table view.
**Impact:** Ketoan can no longer see vehicle plate numbers in the driver list without clicking into each record. For operations involving multiple drivers with the same name or multiple vehicles, this is a workflow friction point.
**Recommendation:** Restore "Biển số xe" column, or make it visible on row expansion. The data still exists in the driver records (verified via add-driver form structure).

---

### UX-14
**Severity:** 🟢 Minor
**Location:** `/accountant/work-orders` — work order card route field
**Observation:** All 21 work order cards in the Ghép chuyến panel show "—" for the route (the text between `·` and the driver name). The route (Điểm lấy → Điểm trả) is available in the suggestion pane (right panel) but not in the card summary on the left list.
**Impact:** Ketoan cannot quickly scan which trips are which route from the left panel. Every trip shows "Công ty TNHH X · —". With 21 trips all from similar customers and routes, visual differentiation is poor.
**Recommendation:** Populate the route summary from `work_order.pickup_location → work_order.dropoff_location`. This is the same data already shown in the right pane's match criteria.

---

### UX-15
**Severity:** 🔵 Polish
**Location:** `/director` — KPI card layout
**Observation:** The stats row shows `35 Tổng chuyến`, `0 Đã khớp`, `17 Chờ xử lý` as separate elements but the visual grouping is unclear. The number "0" for "Đã khớp" appears as a legitimate value rather than an error state, making it hard to distinguish "actually zero" from "data not loaded."
**Impact:** Director dashboard data credibility suffers when legitimate zeros are visually identical to loading/error states.
**Recommendation:** Add a subtle loading skeleton or an error indicator when KPI data fails to load. Consider a "last updated" timestamp on each KPI card.

---

### UX-16
**Severity:** 🔵 Polish
**Location:** `/accountant/settings/clients` — Khách hàng count
**Observation:** The database has been reduced from 17 customers to 5. While this likely reflects a data cleanup, the table previously had a row-count indicator ("17 khách hàng" style). The current build doesn't show a row count in the page header, making it unclear how many records exist.
**Impact:** Minor — but a row count chip ("5 khách hàng") would match the pattern established on other settings pages (Tài xế shows "3 tài xế").
**Recommendation:** Add a row-count chip in the page header consistent with the Tài xế page pattern.

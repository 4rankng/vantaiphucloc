# UI audit — 2026-05-08

Audited at https://phucloc.tingting.vip with the four roles requested. The browser viewport reported by the page during the audit was ~947 × 963 (below the `lg:` breakpoint of 1024), so most pages rendered in mobile/narrow layout. Findings about top-bar/bottom-tab behaviour reflect that mobile path; pure-desktop sidebar issues were verified by reading the source.

## Issues

### Visual — broken images

**V1. `/icons/icon_*.png` 404 on production (HIGH).** Every `PageHeader` with an `icon` prop and the four Director-dashboard quick-action cards render as broken-image placeholders. JS check confirmed the assets respond as `naturalWidth=0` (404). The PNGs exist in `frontend/public/icons/` (committed in `ad7d25c add new icons`, 2026-05-03) and Nginx is configured to serve `*.png`, but the deployed image was built before that commit. **Fix: rebuild + redeploy the frontend image — no code change required.**
- Affected (sample): `/superadmin` ("Tổng quan" header), `/director` ("Truy cập nhanh" cards × 4 + page header), `/director/users`, `/director/partners`, `/director/routes`, `/director/trips`, `/director/notifications`, `/accountant/trips`, `/accountant/work-orders`, `/accountant/partners`, `/accountant/routes`.

### Layout

**L1. Director create-trip — primary submit buttons clipped by bottom tab bar (HIGH).** On `/director/create-trip` (mobile width), the "Hủy" / "Tạo chuyến" footer overlaps the 56-pixel fixed bottom tab bar; the bottom edge of "Tạo chuyến" gets covered. `DirectorLayout` already adds `paddingBottom: 56px` to `<main>`, but the form footer sits in absolute terms within the page-container with extra spacing that pushes part of it below the safe-area. Same risk on `/accountant/create-trip` (uses the same form component).

### UX / functional

**U1. SuperAdmin user dropdown shows leading "@" with no username (MED).** Header dropdown shows `@· SuperAdmin`. The admin seed user has `username` empty/null and the JSX always renders `@${user.username}`. Fix: render `@username` only when present.

**U2. SuperAdmin "Thông tin cá nhân" menu item is disabled (MED).** The route `/superadmin/profile` exists in `router.ts` but the dropdown item in `SuperAdminLayout.tsx` is hardcoded `disabled` — the user has no way to reach their own profile. Fix: enable it and wire `onClick` to `navigate('/superadmin/profile')`.

**U3. Director user-list card shows orphan "·" separator when phone is missing (LOW).** On `/director/users` the "Test Driver" entry renders as `· 99T-99999`. `UserManagement.tsx` lines 241-246 render the `·` separator unconditionally before tractor-plate / CCCD even when no preceding field exists. Fix: only render the `·` if a previous field actually rendered.

### Sidebar / nav — code-level findings

**N1. SuperAdminSidebar is dead code with broken hrefs (LOW).** `frontend/src/components/shared/SuperAdminSidebar/SuperAdminSidebar.tsx` defines nav items pointing to `/superadmin/users` and `/superadmin/notifications`, but the router only declares `/superadmin` (index) and `/superadmin/profile`. The component is never imported anywhere except its own `index.ts`, so users do not see those broken links — but the file is misleading code. Either delete the component or wire it up properly. (Listed as **needs decision**.)

**N2. Bottom mobile-tab labels truncate on the active state (LOW).** "Thông báo" → "Thông b…" on the Director bottom nav when the tab is active (the active style is wider/heavier). Cosmetic. Not fixed in this pass.

### Needs decision (not fixed in this pass)

- N1 (above): keep / delete / wire up `SuperAdminSidebar`?
- The `giamdoc` / `gd_test` / `kt_test` / `ketoan` user-naming inconsistency. Should test users be normalized to the documented `giamdoc`, `ketoan`, `taixe1` triple?
- Driver pages (`/driver/history`, `/driver/notifications`) on a wide-but-narrow viewport (mobile-first centered column) leave a large empty area on the left half. Intentional mobile-only design or should desktop fall back to something else?

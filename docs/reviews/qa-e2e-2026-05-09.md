### F-02 — Driver Home crashes with `splitRouteParts is not defined`

**Type:** Bug
**Layer:** Frontend
**Affected Role/Flow:** Tài xế — DriverHome (`/driver`)
**Description:** Logging in as `taixe`, the home page renders an error fallback "Đã xảy ra lỗi — splitRouteParts is not defined". The driver cannot see their day, current job, or earnings. Only `/driver/history` and `/driver/profile` are reachable by typing URLs directly.
**Severity:** High
**Reproduce:**
1. Login `taixe / admin123`
2. Land on `/driver`
3. Observe red error card

**Evidence:**
- Console: `ReferenceError: splitRouteParts is not defined at d (https://phucloc.tingting.vip/assets/WorkOrderCard-CpTFcl2J.js:1:2157)` → `[ErrorBoundary:page] DriverHome`
- Source bug: `frontend/src/components/shared/WorkOrderCard/WorkOrderCard.tsx:118` calls `splitRouteParts(wo.route)` but the file only imports `{ resolveRoute }` from `@/lib/route-utils`. In `frontend/src/lib/route-utils.ts:3` `splitRouteParts` is declared but **not exported** (only `resolveRoute` is). Bundled output therefore has no binding for it.
- One-line fix candidate: export `splitRouteParts` from `route-utils.ts` and import it in `WorkOrderCard.tsx`, or refactor `DriverCard` to use `resolveRoute` like `AccountantCard` does.

### F-05 — Container number validation is correctly strict but error copy is misleading on check-digit failures

**Type:** Usability Issue
**Layer:** Frontend
**Affected Role/Flow:** Tài xế — Tạo chuyến
**Description:** Entering `TGHU1881778` (real format, but with an incorrect ISO 6346 check digit) shows "Số container sai, xin kiểm tra lại". Entering `QATX1234567` (also 4 letters + 7 digits) shows "Cần 4 chữ cái + 7 số (vd: MSKU1234567)" — but the input *is* 4 letters + 7 digits. Drivers staring at a valid-looking string and an error claiming "needs 4 letters + 7 digits" will be confused. The form is silently rejecting on check-digit, but messaging it as a format problem.
**Severity:** Medium
**Reproduce:**
1. Open `/driver/work-orders/new`
2. Type `QATX1234567` → "Cần 4 chữ cái + 7 số"
3. Type `TGHU1881778` (note: digits are random, not a real check digit) → "Số container sai, xin kiểm tra lại"

**Evidence:** Screenshots showing both error states. Error messaging should distinguish "Định dạng sai" (format) vs "Số kiểm tra sai" (check digit).

### F-06 — Manual `Ghép` allows arbitrary low-confidence matches with no warning

**Type:** Usability Issue / Data-quality Risk
**Layer:** Frontend (and Backend - no server-side guardrail)
**Affected Role/Flow:** Kế toán — Ghép chuyến
**Description:** Tested W001032 ↔ T002009: scored 2/6. The match has:
- Date off by 11 days (`2026-05-09` vs `2026-04-28`)
- Route different (`HẢI AN → Nam Phát, Vietfracht, Northfreight, Sao Đỏ` vs `HẢI AN → ECS, Phương Đông, Tasa Minh Thành…`)
- Container completely different and different size/type (F20 vs E20)

Clicking "Ghép" succeeded with `POST /api/v1/reconcile → 200` and a "Thành công · Đã ghép chuyến" toast. No "Bạn có chắc?" dialog, no second-factor confirmation, no audit-friendly mismatch summary. This will be the dominant source of bad reconciliations once volumes grow.
**Severity:** High
**Reproduce:**
1. Login `ketoan / admin123`
2. `Ghép chuyến` → click any 2/6 trip
3. Pick top suggestion → click `Ghép`
4. Observe success toast with no warning

**Evidence:** Screenshot of pre-match score card + post-match toast. Network log: `POST /api/v1/reconcile → 200`.
**Recommendation:** Show a confirm dialog when score < e.g. 5/6 listing the mismatched fields, with a free-text reason that gets logged (`audit.action='MATCH'` already exists; surface the reason in the toast/audit). Backend should also enforce a `min_confidence` flag.

### F-08 — None of the 11 "chờ khớp" trips reach a 1.0 score, so auto-match is effectively a no-op

**Type:** Bug / Data Issue
**Layer:** Both
**Affected Role/Flow:** Kế toán — Ghép chuyến
**Description:** All 11 unmatched trips on the test env score either 0/6 or 2/6. The 2/6 matches always come from "Khách hàng" + "Điểm lấy" alone. Container, route detail, and date don't agree on any trip. With `FULL_MATCH_THRESHOLD = 1.0`, auto-match cannot actually match anything in this seed. Either (a) the seed is unrealistic, or (b) the matcher's notion of "route" is too literal — it compares the full multi-stop string instead of resolving stops to canonical IDs.
**Severity:** Medium
**Reproduce:** Login ketoan → Ghép chuyến → click "Tự động ghép" → observe count unchanged.
**Evidence:** Network log shows `POST /api/v1/reconcile/auto-match → 200`, but the list count stays at 10.
**Recommendation:** Either tune the seed so a few trips truly are full-match, or relax the threshold to `>= 5/6` (4 weighted criteria) for auto-match and add the audit-trail flag.

### F-12 — Auto-match across alias variants depends entirely on `LocationAlias.status='MATCHED'`, which has no surfaced approval workflow

**Type:** Bug / Missing Feature
**Layer:** Backend + Frontend
**Affected Role/Flow:** Kế toán — Ghép chuyến (alias edge cases)
**Description:** The alias system uses two states: **PENDING** (newly created on import) and **MATCHED** (confirmed equivalent). The matcher considers two locations equivalent when they share a MATCHED alias (`_load_alias_groups` in `match_suggester.py`). On import, `location_resolver.py` *creates* aliases as **PENDING** by default; they only flip to MATCHED when explicitly confirmed. **There is no UI surfaced (in any role I tested) to review and confirm PENDING→MATCHED aliases.** Net effect: alias variants seen for the first time will accumulate as PENDING and silently NOT be considered equivalent for auto-match, which is exactly the symptom the brief was looking for.
**Severity:** High
**Reproduce:** Code review of:
- `backend/app/contexts/operations/infrastructure/match_suggester.py` (`_load_alias_groups`)
- `backend/app/contexts/customer_pricing/infrastructure/location_resolver.py` (default `alias_status = "PENDING"`)
- Frontend search: no `/locations/aliases` route, no admin section, no toast about pending aliases on import.

**Evidence:** Code excerpts above. UI search returned 404 on `/admin/aliases`, `/superadmin/locations`, etc.
**Recommendation:** Surface PENDING aliases in either the kế toán "Cài đặt" or admin section with a one-click "Xác nhận / Từ chối" to promote to MATCHED. Or, on Excel import, batch-prompt a one-page confirmation of new alias mappings before commit.

## Backend alias mapping assessment

**Table exists?** Yes — `location_aliases` with columns `id, location_id, alias, alias_normalized, source, status, created_at, …`.

**Implementation:**
- `match_suggester._load_alias_groups()` builds an equivalence dict `{location_id: {equivalent_ids}}` by reverse-indexing all `LocationAlias` rows where `status='MATCHED'`. Two locations sharing **any** matched alias text become equivalent.
- During match scoring, `pickup_location` and `dropoff_location` criteria call back into this group when comparing trip and order locations, so the location score is alias-aware.
- Import path (`customer_pricing/infrastructure/location_resolver.py`) auto-creates new alias rows as `PENDING` when an unfamiliar string lands. MATCHED requires either a programmatic flip on first sight (the resolver does promote some to MATCHED inline if the location truly exists by name) or a manual admin action.

**Coverage of common aliases (HCM/Sài Gòn/Saigon, HPH/Hai Phong/haiphong, etc.):** Cannot enumerate because (a) the test env has no alias-management UI, and (b) the API requires a session that wasn't reachable from a dangling fetch in the current tab. Code review shows the *mechanism* covers any alias the operator confirms — there is no built-in seed list for common Vietnamese port aliases.

**Recommendation:**
1. Ship a "Địa điểm & bí danh" page (kế toán level) listing all PENDING aliases with one-click confirm/reject (promote to MATCHED).
2. Seed-script a small canonical set: `HCM/SaiGon/Saigon/HCMC`, `HaiPhong/HPH/HaiPhong`, `DaNang/DAD`, etc. — matched at install time so day-1 imports auto-match.
3. On Excel import, return a count of "X new aliases pending review" in the success toast, linking to the new page.
